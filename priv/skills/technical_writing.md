# Technical Writing

When helping users write technical documentation:

## Core Principles

1. **Clarity over cleverness** - Use simple, direct language
2. **Audience awareness** - Know your reader's technical level
3. **Scannability** - Use headings, lists, and formatting
4. **Accuracy** - Verify technical details
5. **Completeness** - Include all necessary information

## Document Types

### README Files
Structure:
1. **Project name and one-line description**
2. **Installation** - How to get started
3. **Usage** - Basic examples
4. **Configuration** - Available options
5. **Contributing** - How to help
6. **License** - Legal information

### API Documentation
For each endpoint or function:
- **Description** - What it does
- **Parameters** - Name, type, required/optional, description
- **Return value** - Type and structure
- **Examples** - Request and response
- **Errors** - Possible error conditions

### Tutorials
Structure:
1. **Goal** - What the reader will learn
2. **Prerequisites** - What they need to know/have
3. **Steps** - Numbered, actionable instructions
4. **Verification** - How to confirm success
5. **Next steps** - Where to go from here

## Writing Guidelines

### Be Concise
- Cut unnecessary words
- One idea per sentence
- Active voice over passive

### Be Specific
- "Click the blue Submit button" not "click the button"
- Include version numbers and specific values
- Provide exact commands to run

### Be Consistent
- Use the same terms throughout
- Follow a style guide
- Maintain consistent formatting

## Code Examples

- Keep examples minimal but complete
- Show expected output where relevant
- Highlight the key parts
- Test that examples actually work
